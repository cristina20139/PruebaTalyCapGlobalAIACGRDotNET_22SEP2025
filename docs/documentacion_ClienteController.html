📘 ClienteController

Controlador REST que expone operaciones sobre Cliente.

🏗️ Rol en la Arquitectura Hexagonal

Este controlador pertenece a la capa de infraestructura y actúa como adaptador de entrada:

  • 🌐 Recibe solicitudes HTTP.
  • ➡️ Delegar la lógica de negocio a ClienteService.
  • 🔌 Se integra con la aplicación sin acoplarla a frameworks.
  • 🛡️ Favorece el desacoplamiento entre la infraestructura y el dominio.
📌 Nota: En la arquitectura hexagonal, los controladores REST son “puertos de entrada” que permiten a sistemas externos interactuar con el núcleo de la aplicación.

✨ Principios aplicados

  • 📏 Single Responsibility (SRP): expone endpoints y delega la lógica de negocio.
  • 🔄 Dependency Inversion (DIP): depende de la abstracción ClienteService, no de implementaciones concretas.
  • 🧩 Open/Closed Principle (OCP): se pueden agregar endpoints sin alterar la lógica actual.
  • 📝 Logs claros: info, advertencias ⚠️ y errores ❌ para trazabilidad.

⚙️ Manejo de Errores

Los errores se manejan con @RestControllerAdvice y devuelven códigos HTTP adecuados:

  • ✅ 200 OK: Cliente encontrado.
  • ⚠️ 400 Bad Request: Parámetros inválidos.
  • ❌ 404 Not Found: Cliente no encontrado.
  • 💥 500 Internal Server Error: Error inesperado.

📥 Método destacado

🔍 obtenerCliente

Busca un cliente por tipo y número de documento.

  • Parámetros:
    • 🆔 tipoDocumento: Tipo de documento (CC, CE, Pasaporte).
    • #️⃣ numeroDocumento: Número de documento (> 0).
  • Retorna: ResponseEntity<Cliente> con el cliente o error.
  • Excepciones:
    • ⚠️ IllegalArgumentException: parámetros inválidos.
    • ❌ NoSuchElementException: cliente no encontrado.
    • 💥 RuntimeException: error inesperado.
