📄 Documentación de la interfaz ClienteRepository

📌 Rol en la Arquitectura Hexagonal

La interfaz ClienteRepository actúa como un puerto de persistencia. 👉 Define las operaciones que la capa de dominio espera, sin importar cómo se implementen.

  • 🔌 Puerto: separa la lógica de negocio de la infraestructura.
  • 🗄️ Adaptadores: pueden implementarse con JDBC, JPA, MongoDB, APIs externas, etc.
  • 🔄 Independencia: el servicio de aplicación depende solo de esta interfaz, nunca de detalles técnicos.

🚀 Uso con Spring Boot

Dentro de un proyecto con Spring Boot:

  • 📦 Se puede anotar una implementación concreta con @Repository.
  • 🧩 Gracias a la inyección de dependencias de Spring, los servicios pueden usar esta interfaz sin conocer la implementación.
  • 🗃️ Si se usa Spring Data, incluso puede generarse automáticamente una implementación del repositorio.
👉 Spring Boot proporciona la infraestructura, pero la interfaz sigue siendo puro dominio.

📖 Javadoc en formato legible

Descripción: Puerto de persistencia para operaciones sobre Cliente. Define los métodos que debe implementar cualquier repositorio, manteniendo la independencia entre dominio e infraestructura.

🌟 Principios aplicados

  • ✅ SRP (Single Responsibility): cada método tiene una sola responsabilidad (obtener o insertar clientes).
  • 📐 ISP (Interface Segregation): la interfaz es pequeña y específica.
  • 🔄 DIP (Dependency Inversion): la capa de aplicación depende de esta abstracción, no de implementaciones concretas.

🛠️ Métodos

Optional<Cliente> obtenerCliente(String tipoDocumento, long numeroDocumento)

🔍 Obtiene un cliente por tipo y número de documento.

  • 📥 Parámetros: tipoDocumento (no nulo ni vacío), numeroDocumento (mayor que 0).
  • 📤 Retorno: Optional<Cliente> con el cliente si existe, vacío si no.
void insertarCliente(Cliente cliente)

🆕 Inserta un nuevo cliente en el repositorio.

  • 📥 Parámetro: cliente (no nulo).
