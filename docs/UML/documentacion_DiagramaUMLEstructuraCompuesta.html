Documentación Detallada del Diagrama de Arquitectura Hexagonal (Ports & Adapters)

1. Descripción General

Este documento describe la arquitectura interna del módulo de clientes bajo el patrón Hexagonal (Ports & Adapters) usando un diagrama UML de componentes y dependencias. El diagrama muestra cómo los servicios de aplicación interactúan con los puertos (interfaces) y cómo la infraestructura implementa dichos puertos, manteniendo el dominio desacoplado de detalles tecnológicos.

2. Versión y Especificación UML

  • Versión de UML: 2.5.1
  • Tipo de diagrama: Diagrama de Componentes y Colaboraciones (Component Diagram)
  • Año de creación: 2025
Nota: El uso de UML 2.5.1 permite modelar claramente la separación de capas y dependencias entre dominio, aplicación e infraestructura, siguiendo el enfoque hexagonal.

3. Explicación del Diagrama

  1. Aplicación:
    • ClienteService («service»): Expone los servicios de negocio (obtenerCliente, crearCliente).
    • Depende de un puerto («port») llamado ClienteRepository, que define las operaciones de persistencia esperadas.
  2. Infraestructura:
    • ClienteRepositoryImpl («component»): Implementa el puerto ClienteRepository para acceder a la base de datos.
    • Utiliza JdbcTemplate y ClienteRowMapper como partes internas para ejecutar consultas y mapear resultados.
  3. Dominio:
    • Cliente («entity»): Entidad principal del dominio que representa los datos de un cliente.
  4. Relaciones:
    • La aplicación depende de un puerto, que es implementado por la infraestructura mediante inyección.
    • ClienteRepositoryImpl utiliza composición para interactuar con JdbcTemplate y ClienteRowMapper.
    • El mapeo de datos conecta la infraestructura con las entidades del dominio.

4. Justificación del Uso de Diagrama UML de Componentes (Arquitectura Hexagonal)

Este diagrama es esencial porque:
  • Permite visualizar la separación entre el núcleo de negocio y los detalles de infraestructura.
  • Facilita el mantenimiento, escalabilidad y pruebas, al definir contratos claros (puertos) y dependencias explícitas.
  • Ayuda a identificar responsabilidades y a mantener un sistema desacoplado y flexible ante cambios tecnológicos.
  • Es un estándar útil para comunicar la arquitectura a desarrolladores y arquitectos.

5. Estilo Visual

El documento mantiene un estilo profesional y claro:
  • Colores suaves, bloques diferenciados y tipografía moderna.
  • El diagrama está centrado y segmentado por capas (aplicación, infraestructura, dominio).
  • Las relaciones están etiquetadas (“puerto”, “inyección”, “mapeo”, “parte interna”) para máxima claridad.
