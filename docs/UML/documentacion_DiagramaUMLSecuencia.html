Documentación Detallada del Diagrama de Secuencia UML – Manejo de Excepciones en Consulta de Cliente

1. Descripción General

Este documento describe, mediante un diagrama de secuencia UML detallado, los flujos principales y alternativos (excepciones) durante la consulta de cliente en el sistema. Se modelan los escenarios de éxito, cliente no encontrado, error de validación y error inesperado, mostrando cómo circulan los mensajes y excepciones entre los distintos componentes.

2. Versión y Especificación UML

  • Versión de UML: 2.5.1
  • Tipo de diagrama: Diagrama de Secuencia (Sequence Diagram, con excepciones y ramificaciones)
  • Año de creación: 2025
Nota: El diagrama incluye líneas de vida para todos los componentes relevantes, modelando fragmentos de control y manejo de excepciones conforme a UML 2.5.1.

3. Explicación del Diagrama

  1. Flujo principal: Cliente encontrado
    • El usuario solicita un cliente utilizando GET /clientes/{tipoDocumento}/{numeroDocumento}.
    • La petición fluye por ClienteController, ClienteService, ClienteRepository hasta ClienteRepositoryImpl, que ejecuta un procedimiento almacenado en DB SQL Server.
    • Si el cliente existe, se retorna el objeto y una respuesta HTTP 200 OK.
  2. Flujo alternativo: Cliente no encontrado
    • Si el cliente no existe, el ResultSet está vacío y se retorna Optional.empty.
    • Se lanza NoSuchElementException y el GlobalExceptionHandler retorna HTTP 404 Not Found.
  3. Flujo alternativo: Error de validación
    • Si el número de documento es inválido, se lanza IllegalArgumentException y el handler retorna HTTP 400 Bad Request.
  4. Flujo alternativo: Error inesperado
    • Si ocurre un error inesperado (por ejemplo, error en la base de datos), se lanza RuntimeException y se retorna HTTP 500 Internal Server Error.

4. Justificación del Uso de Diagrama UML de Secuencia con Excepciones

Este diagrama es relevante porque:
  • Permite visualizar flujos de éxito y error en detalle, incluyendo excepciones y su manejo.
  • Facilita la comprensión de la robustez y resiliencia del sistema ante distintas condiciones.
  • Documenta claramente cómo los errores son propagados y gestionados, útil para desarrollo y QA.
  • Establece el contrato de respuestas esperadas para distintos escenarios.

5. Estilo Visual

El documento mantiene un estilo profesional y claro:
  • Colores suaves, bloques diferenciados, tipografía moderna.
  • El diagrama está centrado y segmentado por escenarios para fácil navegación.
  • Se destacan actores, componentes, ramificaciones y excepciones.
